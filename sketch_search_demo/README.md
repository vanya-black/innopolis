# ✏️ Нарисуй — найди похожее

Второе демо к лекции про эмбеддинги (первое — `../game_map_demo`). Человек
**рисует объект** на холсте, а мы находим **визуально похожие фотографии**.
Живая иллюстрация слайда «Поиск по картинкам — найди визуально похожее».

## Идея в одну строку

**CLIP** кодирует и фотографии, и рисунок в **одно** векторное пространство —
значит, рисунок можно искать среди фото по косинусной близости. Та же механика,
что в demo №1, только «запрос» — это рисунок, а корпус — картинки.

Три файла = три шага:

| Шаг | Файл | Что делает |
|-----|------|------------|
| 1. Фото-галерея | `collect.py` | качает датасет **Sketchy** (зеркало `JamieSJS/sketchy` на HF) → `data/photos/<категория>/` |
| 2. Эмбеддинги | `embed.py` | кодирует фото моделью **CLIP** `clip-ViT-B-32` → `artifacts/embeddings.npy` |
| 3. Дашборд | `app.py` | холст (**streamlit-drawable-canvas**) → поиск ближайших фото |

`embed.py` ещё и общий «движок»: дашборд берёт оттуда модель, подготовку рисунка
и поиск соседей.

## Запуск

```bash
cd sketch_search_demo
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python collect.py     # 1) скачать галерею (~260 МБ parquet → 12 500 фото ≈ 200 МБ)
python embed.py       # 2) CLIP-эмбеддинги (на Apple MPS/GPU ≈ пара минут)
streamlit run app.py  # 3) дашборд на http://localhost:8501
```

…или одной командой: `bash run_all.sh`.

Готовый индекс (`artifacts/embeddings.npy` + `gallery.json` + `meta.json`) уже в
репозитории. Сами фотографии (200 МБ) в git не хранятся — их восстанавливает
`collect.py` (детерминированно, индекс остаётся выровненным).

## Данные

- **Sketchy Database** (Sangkloy et al., 2016) — фотографии 125 категорий объектов.
  Берём только фото через HF-зеркало `JamieSJS/sketchy` (parquet 260 МБ) — самый
  лёгкий программный путь, без исходных 3.4 ГБ.
- 25 000 строк = 12 500 уникальных фото × 2 кадрирования; берём один кадр на объект.
- Имя категории восстанавливаем из ImageNet-synset в `id` через `nltk.wordnet`
  (`n02121620` → `cat`).

## Как это работает (для слайда)

1. CLIP заранее перевёл каждую из 12 500 фотографий в вектор из 512 чисел.
2. Рисунок проходит через **ту же** модель и попадает в **то же** пространство.
3. Ищем фото с ближайшими векторами — это и есть «похожие».

Отвечает мгновенно: тяжёлые эмбеддинги посчитаны заранее (`embed.py`), в реальном
времени — лишь один прогон CLIP по рисунку и поиск ближайших.

## Заметки

- Рисуйте **чёрным по белому, крупно и по центру**. Рисунок автоматически
  обрезается по рамке, центрируется и сглаживается (`embed.preprocess_sketch`).
- CLIP ловит **категорию и форму** (кошка → кошки), но не точную позу: рисунок и
  фото — разные «домены» (слайд «где это ломается»). Фото кодируем **как есть**,
  подготовку проходит только рисунок.
- Холст: форк **`streamlit-drawable-canvas-fix`** — оригинал заброшен (2025) и
  падает на свежем Streamlit.
